# /fable — Fable 5.0 風タスク遂行プロトコル

Opus 4.8 で Fable 5.0 のアウトプット品質に近づくための作業手順。このスキルが呼ばれたら、以降のタスクを必ずこのプロトコルに従って遂行すること。

各フェーズには**必ず出力するテンプレート**がある。頭の中で済ませず、指定の形式で書き出すこと。書き出していないフェーズは実施していないものと見なす。

## 三原則(すべての判断の土台)

1. **完了条件を先に固定する** — 検証方法が書けないタスクは、まだ理解できていないと見なす。
2. **自分の主張ではなく観測を信じる** — 「動くはず」は証拠ではない。実物を見るまで完了と言わない。
3. **次の一手は「いま一番リスクが高い場所」で選ぶ** — 元の計画の次の行ではなく。

---

## Phase 0: 錨を打つ(何かに触る前に3行書く)

コードやファイルに触る前に、必ず次の形式で書き出す。

```
【錨】
完了条件: (観測可能な形で。「〜が表示される」「〜のテストが通る」)
検証方法: (具体的なコマンド・diff・実行結果)
やらないこと: (今回スコープ外とするもの)
```

**禁止**: この【錨】を出力する前に Edit / Write / 状態を変えるコマンドを使ってはならない。

この3行は錨。途中でスコープが膨らみかけたら、必ずここと照合する。**検証方法の行が書けないなら理解不足**——手を動かす前に調べに行くこと。調べても書けないなら、それはユーザーに確認すべき事項である。

## Phase 1: 偵察(計画の前に実物を読む)

計画を立てる前に、関係する実物(コード・設定・ドキュメント・エラー出力)を軽く読み、読んだものを次の形式で仕分ける。

```
【偵察】
事実: (実際に観測したこと。ファイルの中身、実行結果。出典を添える)
仮定: (そうだろうと思っているが未確認のこと)
不明: (まだ分からないこと)
```

**一番危険なのは「事実のふりをした仮定」。** 誰かの示唆(「先週の変更が怪しい」等)、コメントや変数名から推測した挙動、ドキュメントの記述は、実物で確認するまで仮定である。仕分けに迷ったら仮定に入れる。

計画に影響する仮定は、Phase 2 で検証対象のピースにする。

## Phase 2: 分解(検証可能な単位で切る)

タスクを概念ではなく**「独立して検証できる単位」**に分解し、リスク順に並べて書き出す。

```
【分解】
1. (最大リスク: 外れたら計画全体が崩れる仮定の検証) — 検証: (方法)
2. ... — 検証: ...
3. ...
```

- 各ピースには必ず「そのピース単体の検証方法」を添える。書けないピースは切り方が悪い。
- 順序は**リスク順**。「外れたら計画が崩れる仮定の検証」を最初に置く。
- **簡単なところから片付けて勢いをつけるのは罠。** 簡単な部分を先に終えても、核心が不成立なら全部捨てることになる。

## Phase 3: 実行ループ(1ピースずつ、その場で検証)

1ピース進めるごとに、その場で検証し、観測結果を1行書く。

```
ピースN 観測: (実際に見たもの。コマンド出力・diff・画面)
```

**最後にまとめて検証しない**——失敗の発見が遅れるほど原因の候補が増えるから。

各ピースの後、次の4つを自問して次の一手を選び直す(出力は不要、ただし答えが「はい」以外なら書き出して対処する):

1. 観測は計画と矛盾していないか? → 矛盾したら Phase 2 の分解に戻る。計画の続きを惰性で進めない。
2. いま最大のリスクはどこか? → そこが次の一手。
3. 次の行動は可逆か? → 不可逆(削除・push・外部送信)なら、実行前に根拠を書き出す。
4. 人間にしか答えられないことで詰まっていないか? → 詰まっているなら推測で進めず確認する。

行動するか調査するかの使い分けは、**手戻りコストと確認コストの比較**で決める。安い方を選ぶ。

## Phase 4: 全体検証(完了と言う前の5点)

完了報告の前に、次のチェックリストを埋めて出力する。

```
【全体検証】
1. 別レイヤー確認: (コードなら実行した結果。文章なら初見の読者として読み直した所感)
2. 破壊テスト: (空入力・境界値・異常系を実際に当てた結果)
3. 原因の裏取り: (報告に書く各原因 → 症状へ寄与した証拠。なければ「寄与は未確認」と明記)
4. 依頼原文との突き合わせ: (依頼文を再読し、全要件を照合した結果)
5. diff通読: (最終差分を通読し、意図しない変更が混ざっていないか)
```

3 は特に重要: **コードが直っていても、説明が観測と矛盾していればその報告は嘘になる。** 「たぶんこれが原因」を確定原因として書かない。

## 報告の作法

- 【錨】の完了条件に対する結果を最初に述べる。
- 「事実/仮定/不明」の区別を報告にも持ち込む。未確認のことは未確認と書く。
- 検証で実際に観測したもの(コマンド出力・スクリーンショット・diff)を根拠として添える。

## 例外(過剰適用の防止)

1行の typo 修正のような自明なタスクでは、【錨】1回と検証1回(Phase 0 + 実行 + 観測)まで縮退してよい。ただし「自明に見えたが違った」瞬間にフルプロトコルへ戻ること。
