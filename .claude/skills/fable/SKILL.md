# /fable — Fable 5.0 風タスク遂行プロトコル

Opus 4.8 で Fable 5.0 のアウトプット品質に近づくための作業手順。このスキルが呼ばれたら、以降のタスクを必ずこのプロトコルに従って遂行すること。

## 三原則（すべての判断の土台）

1. **完了条件を先に固定する** — 検証方法が書けないタスクは、まだ理解できていないと見なす。
2. **自分の主張ではなく観測を信じる** — 「動くはず」は証拠ではない。実物を見るまで完了と言わない。
3. **次の一手は「いま一番リスクが高い場所」で選ぶ** — 元の計画の次の行ではなく。

---

## Phase 0: 錨を打つ（何かに触る前に3行書く）

コードやファイルに触る前に、必ず次の3行を書き出す。

- **完了条件**: 観測可能な形で（「〜が表示される」「〜のテストが通る」）
- **検証方法**: 具体的なコマンド・diff・実行結果
- **やらないこと**: 今回スコープ外とするもの

この3行は錨。途中でスコープが膨らみかけたら、必ずここと照合する。**検証方法の行が書けないなら理解不足**——手を動かす前に調べに行くこと。

## Phase 1: 偵察（計画の前に実物を読む）

計画を立てる前に、関係する実物（コード・設定・ドキュメント）を軽く読み、読んだものを3つに仕分ける。

- **事実**: 実際に観測したこと（ファイルの中身、実行結果）
- **仮定**: そうだろうと思っているが未確認のこと
- **不明**: まだ分からないこと

**一番危険なのは「事実のふりをした仮定」。** 誰かの示唆（「先週の変更が怪しい」等）は仮定であって事実ではない。仕分けに迷ったら仮定に入れる。

## Phase 2: 分解（検証可能な単位で切る）

タスクを概念ではなく**「独立して検証できる単位」**に分解する。

- 順序は**リスク順**。「外れたら計画全体が崩れる仮定の検証」を最初に置く。
- **簡単なところから片付けて勢いをつけるのは罠。** 簡単な部分を先に終えても、核心が不成立なら全部捨てることになる。

## Phase 3: 実行ループ（1ピースずつ、その場で検証)

1ピース進めるごとに、その場で検証する。**最後にまとめて検証しない**——失敗の発見が遅れるほど原因の候補が増えるから。

各ピースの後、次の4つを自問して次の一手を選び直す:

1. 観測は計画と矛盾していないか？
2. いま最大のリスクはどこか？
3. 次の行動は可逆か？
4. 人間にしか答えられないことで詰まっていないか？

行動するか調査するかの使い分けは、**手戻りコストと確認コストの比較**で決める。

## Phase 4: 全体検証（完了と言う前の5点）

1. **作った層と別の層で確認する** — コードなら実行する。文章なら初見の読者として読む。
2. **壊しに行く** — 空入力・境界値を実際に当てる。
3. **原因の説明を観測で裏取りする** — 報告に書く各原因に「症状へ寄与した証拠」を1つずつ持つ。なければ「寄与は未確認」と明記する。コードが直っていても、説明が観測と矛盾していればその報告は嘘になる。
4. **依頼原文との突き合わせ** — 依頼文をもう一度読み、全要件が満たされているか照合する。
5. **diff通読** — 最終的な差分を通しで読み、意図しない変更が混ざっていないか確認する。

---

## 報告の作法

- 完了条件（Phase 0 の3行）に対する結果を最初に述べる。
- 「事実/仮定/不明」の区別を報告にも持ち込む。未確認のことは未確認と書く。
- 検証で実際に観測したもの（コマンド出力・スクリーンショット・diff）を根拠として添える。
